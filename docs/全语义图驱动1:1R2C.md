# React 全语义图驱动 1:1 R2C 闭环计划（修订版，按现仓代码可执行）

这份《React 全语义图驱动 1:1 一次生成闭环计划》是一份极具魄力、极其硬核的架构重构 RFC。你精准切中了当前系统腐化的根源：**基于字符串的魔法扫描、C 层的硬编码补洞、分离的真值源以及缺失的闭环门禁**。

选择破釜沉舟，全面拥抱“AST 静态分析 + 图驱动 AOT 编译 + 严格一次门禁”，是彻底解决 UI 漂移和跨端一致性问题，带领 R2C 走向工业级编译器的**唯一正途**。

然而，站在现代 UI 编译器（如 React Forget, SolidJS, Svelte）和大型端到端测试基建的视角来看，这份实施计划（Implementation Plan）在**“动静结合边界”、“AOT 范式选择”以及“开发者体验 (DX)”**上存在几个致命的架构陷阱。如果不加以修正，你的“全路由一次门禁”大概率会变成永远无法跑通的死局。

以下是深度的隐患剖析、核心改进点以及更优的架构升级方案：

---

### 🚨 一、 计划中隐藏的致命问题与逻辑悖论

#### 1. 静态 AST 与动态 Runtime Trace 强行合并的“逻辑悖论”（针对 Step 5、6、7）

* **问题所在**：AST 包含了代码的**全量静态可能性**（如 `if (loading) <Skeleton/> else <List/>`）。而 Runtime Trace（计算样式、Layout Box）提取的是**某一特定状态瞬间的动态结果**。
* **致命后果**：如果你强行将 Runtime Trace 合并进 AST 生成“单一语义总图”，AST 中的大量备用分支节点在 Trace 中会找不到对应的样式和布局数据，直接触发你设定的“冲突直接编译失败”。**这意味着任何带有条件渲染的组件都无法通过编译。**

#### 2. “绝对布局数值 (Layout Box)”破坏了跨端响应式（针对 Step 6、15）

* **问题所在**：计划要求将运行时采集的 `layout box`（物理坐标 x, y, width, height）作为样式真值，并同构复制到 iOS/Harmony。
* **致命后果**：Android 跑出的绝对像素值，下发到屏幕比例、DPI 完全不同的 iOS 设备上，会导致毁灭性的排版错乱。把“绝对数值”当做“跨端真值”写入产物，等于把跨端框架做成了死板的图片热区，彻底丧失了 Flexbox 的弹性自适应能力。

#### 3. 架构缝合怪：AOT 细粒度指令 vs 重度 React 运行时（针对 Step 8、9）

* **问题所在**：Step 8 要求 AOT 产出组件级 `mount/update` 执行单元（这是类似 SolidJS 的无 VDOM 细粒度范式）；而 Step 9 又要求维持 `render -> commit -> layout` 的 React 调度语义（重度 VDOM 范式）。
* **致命后果**：两者水火不容。既然在编译期（AOT）已经算出了更新指令，就不该再引入昂贵且复杂的 Fiber 调和引擎。如果在 Cheng 中手写 React 运行时，不仅极易引发时序 Bug，**“Hook 槽位错乱”**的幽灵将永远无法根除。

#### 4. “盲回放”导致门禁极度脆弱（Flakiness）（针对 Step 3、14）

* **问题所在**：依赖解析 `actions[]` 的硬指令盲目回放，并要求全路由 `framehash` 绝对一致。
* **致命后果**：真实的业务有网络请求、Suspense 和动画。盲目的点击极易打在未渲染完的骨架屏上，网络延迟的微小波动会导致后续的 `framehash` 永远对不齐。CI 的假阳性报错将淹没团队。

#### 5. 全路由黑盒硬门禁带来的 DX 灾难（针对 Step 14）

* **问题所在**：全量路由一次跑通，中间失败直接 Hard-fail，不降级。
* **致命后果**：如果有 50 个路由，流水线跑到第 48 个页面时因为 1px 的误差崩溃。开发者毫无上下文，无法断点，没有中间态，排查问题犹如大海捞针。

### 💡 二、 核心改进点 (Actionable Improvements)

为了保住你“全语义、无补丁、1:1”的战略目标，必须对编译管线做以下关键修正：

#### 改进 1：解绑“编译期真值”与“门禁校验真值” (重构 Step 7)

* **不要把 Runtime Trace 合并回 AST 编译图中！**
* **编译期**：`compiler.cheng` 通过 AST 提取完整的 Flexbox 规则和 CSS 变量，直接生成跨端的静态布局指令。
* **校验期**：Runtime Trace 只作为 `verify_r2c_equivalence` 的 **测试神谕 (Test Oracle)**。Cheng 运行时利用内置的轻量布局引擎（如 Yoga）根据当前屏幕算出布局，然后去和 Trace 的 Layout Box 对比误差。**验证通过即可，不写入产物。**

#### 改进 2：路由动作图必须引入“状态屏障” (重构 Step 3)

* 废弃单纯的点击序列，`r2c_route_actions_v1.json` 必须升级为**基于意图的状态机**。
* 增加前置断言：`{ "action": "click", "target_id": "btn", "wait_for_condition": "network_idle", "expect_state": "modal_open" }`。执行器必须等待底层引擎 Settle（渲染就绪）后才执行动作和截图，彻底消除异步时序导致的门禁失败。

#### 改进 3：强制注入“绝对确定性沙箱” (补充 Step 10/14)

* 在跑全路由门禁和提取 Truth Trace 时，必须在 C 层的 `webapi` 桥拦截并注入**绝对一致的 Mock 数据**。彻底冻结时间戳（固定 `Date.now`），并强行关闭所有过渡动画。保证同一份代码跑 100 次，生成的树和 Hash 100% 一致（幂等性）。

#### 改进 4：配备“外科手术式”的快照恢复机制 (优化 Step 14)

* 坚持全路由门禁，但一旦发生 Diff 失败，执行器必须自动 Dump 出：
1. 失败那一刻的 `Route_State_Snapshot.json`。
2. 可视化的 Truth vs Actual 布局差异高亮图。


* 提供本地验证命令：`verify --resume-from-snapshot [id]`。允许开发者直接起一个隔离沙箱，从失败的那个弹窗状态开始单步调试，直到修绿后再推 CI。

---

### 🚀 三、 更好的终极架构方案：响应式 AOT 降维打击

如果你已经决心重写 `codegen` 并且要把 C 层压缩到最小 ABI，强烈建议彻底放弃模拟 React 的路线，直接执行**降维打击**：

#### 方案：SolidJS / Svelte 风格的“细粒度响应式编译”

1. **废弃 `react_compat.cheng` 的宏观调度**：不再模拟庞大的 Fiber 树、`render -> commit` 和虚拟 DOM 链表。
2. **前端深度分析**：在 `compiler.cheng` 的 AST 阶段引入**数据流依赖追踪 (Dependency Tracking)**。精准识别哪个 `useState` 影响了哪个具体的 DOM 属性。
3. **极简执行单元 Codegen**：将 React 语义直接 Lowering 为绑定了底层信号 (Signals) 的闭包。
* *传统运行时*：状态变化 -> 执行组件产出新树 -> Diff -> 更新。
* *你的新 AOT*：分析出 `count` 绑定在 `<Text>` 上，直接生成 C/Cheng ABI 指令：`if (count_changed) { TextNode_12.setText(count); }`。


4. **最终收益**：彻底消灭了 Hook 槽位引发的疑难杂症，因为 AOT 阶段已经把 Hook 拍扁成了具体的更新指令。C 层的 ABI 真正退化为类似 GPU Command Buffer 的纯 UI 指令接收器，没有任何业务路由逻辑。这会让你的 Step 11 和 Step 15 变得极其轻松，且运行性能全面碾压原生 RN。

### 总结

你的“一刀切门禁、全图驱动、干掉补丁”是治愈系统架构级顽疾的猛药，方向无可挑剔。只要处理好**“静态编译规则”不强融“动态布局数值”**，将运行时快照回归到“测试基准”的本位，同时借助响应式 AOT 彻底甩掉 React VDOM 负担，这套架构将完全具备统治级的稳定性和极高的性能天花板。

## Implementation Plan
### Phase 0: 契约冻结（先做）
1. 先以 `native_r2c_report_validate.c` 为唯一真值源冻结 compile report 契约，文档与 gate 同步。
2. 保持 `native_r2c_compile_react_project.c` 只做编译 orchestrator，不新增任何“编译后补写产物”逻辑。

### Phase 1: 路由数据化（低风险先收口）
3. 在 `src/tools/native_capture_android_unimaker_truth.c` 下线 `route_parent_for/route_depth_for` 硬编码路径，改为读取 `route_tree_path` 生成父子关系和 depth。
4. 在 `src/tools/native_capture_route_layer_android.c` 将回放改为“先读 `actions[]`，缺失时回退 `action_script` 并告警”：
   1. `actions[]` 支持 `launch_main/sleep_ms/tap_ppm/keyevent`。
   2. 回放结果必须回传命中目标 `route_state`，否则 fail。
5. 在 `src/tools/r2c_aot/compiler.cheng` 继续输出 `actions[] + action_script` 双字段一段时间，确保旧链路兼容。

### Phase 2: 编译前端升级（分步替换）
6. 在 `compiler.cheng` 逐步下线 `cpCollect*`、`detectTextContains` 等扫描逻辑，优先复用现有 `tsx_extract.cheng/jsx_lower.cheng/module_graph.cheng/ir.cheng`，避免从零重写。
7. AST/IR 产物最小闭环：
   1. 结构图：组件边界、节点层级、模块依赖。
   2. 行为图：事件绑定、hook 槽位、effect 依赖、context 订阅。
   3. 路由图：route tree、actions、path_from_root。
8. 编译阶段产出统一 `semantic_graph`，并保持节点 ID 稳定（用于 framehash 对齐和回归定位）。

### Phase 3: 运行时图融合与代码生成
9. 增加 runtime 轨迹采集工具（C 或 Cheng 实现均可），按路由采集 `computed style/layout box/event target map` 到 `style_graph/runtime_trace`。
10. 合并规则固定：`AST/IR 决定结构与语义；runtime trace 决定最终样式数值`；冲突 hard-fail。
11. 重写 `codegen_cheng.cheng` 与 `runtime_generated_template.cheng`，输出组件级 `mount/update/unmount`，禁止文本语义节点回放兜底。
12. 完善 `react_compat.cheng` 生命周期语义与 `webapi.cheng` 副作用桥协议（render 阶段禁副作用）。

### Phase 4: C 层瘦身与严格门禁
13. 在 `cheng_mobile_exports.c` 逐步下线路由文字绘制和手写 route 切换，仅保留系统 ABI（输入/生命周期/副作用桥/帧采集）。
14. 首页硬门禁保持：`route_state=home_default` 不一致直接失败，禁止静默漂移。
15. Android 全路由门禁输出必须包含：`route + action step + runtime_state + truth diff`。
16. Android 全绿后同构到 iOS/Harmony，门禁标准不降级。

## Better Plan（推荐落地节奏）
1. 第一步只做“契约收口 + 路由数据化”（Phase 0 + Phase 1），先把 gate 变成可定位、可复现。
2. 第二步再做“前端升级 + 图融合”（Phase 2 + Phase 3），每次只替换一条链路并保持旧路径可回退。
3. 第三步执行“C 层瘦身 + 严格删除兼容路径”（Phase 4），在 Android 全绿稳定后删除 `action_script` 消费与硬编码 route 逻辑。

## Assumptions And Defaults
1. 采用“全路由一次门禁”，不做分层放行。
2. 样式真值源固定为运行时计算样式图，不接受纯静态 className 作为最终真值。
3. 第三方未映射能力维持“未映射即失败”，不做静默 stub。
4. 文档不绑定机器绝对路径；构建路径通过环境变量和仓库相对路径解析（例如 `CHENG_TOOLING_BIN`、`PKG_ROOTS`）。
5. C 层仅保留 ABI 桥能力；业务语义、路由和组件渲染由 Cheng 产物执行。
6. 最终收口由 `verify_r2c_equivalence_all_native` 与 `verify_production_closed_loop` 完成。
